## Findings & Recommendations (for later fixing)

These did NOT fail the run but are defects/risks the test surfaced:

1. **Server drops request body on segmented TCP writes (robustness bug).**
   The hand-rolled HTTP parser reads the body from a single socket read. A
   client that sends headers and body in separate TCP segments (Python
   `urllib`, some HTTP libs, slow networks) gets `Invalid JSON: EOF while
   parsing`. Browser `fetch` and `curl` coalesce small bodies so it's hidden
   today. Fix: loop-read the socket until `Content-Length` bytes are received
   before parsing. Priority: medium (breaks non-browser API clients / future
   integrations).

2. **No backend global container cap per call.** Per-warehouse capacity is
   enforced (409 when loaded ≥ planned), but total containers across a call
   is only capped in the frontend (skill pitfall #57). A direct API caller
   can exceed `total_containers`. Fix: add a call-level count check in
   `handle_create_container`. Priority: low (single-org, shared token).

3. **No test-data teardown.** This script only creates; repeated runs
   accumulate rows (DB now holds prior-run leftovers). Recommend a
   `--cleanup` flag or a SQL reset for repeatable CI. Priority: low.

4. **201 vs 200 inconsistency (cosmetic).** Creates return 201, updates 200
   — correct REST, but verify all frontend callers treat 2xx uniformly (they
   do via `res.ok`). No action needed; noted for API-doc accuracy.
